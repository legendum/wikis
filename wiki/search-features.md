# Search Features

The "wikis" project provides full-text search over wiki pages using a hybrid approach that combines SQLite FTS5 for fast keyword matching with Retrieval-Augmented Generation (RAG) re-ranking via vector embeddings. This design enables efficient querying of large wikis by first retrieving candidate chunks quickly via lexical search, then re-ranking them for semantic relevance using cosine similarity on embeddings. Search operates exclusively on chunks of wiki content stored in the `wiki_chunks` table and its FTS5 virtual table `wiki_chunks_fts`. For details on the database schema, see [Database Storage](database-storage.md).

## Overview

Search retrieves relevant chunks from wiki pages, returning the file path, chunk content, and a relevance score between 0 and 1. The process unfolds in two stages:

1. **FTS5 Candidate Retrieval**: SQLite's FTS5 performs keyword matching on up to 50 candidates (`FTS_CANDIDATES = 50`), ordered by BM25 rank. Queries are escaped to handle special characters safely by quoting words and supporting prefix wildcards (`*`).
2. **Vector Re-ranking**: The query is embedded using Ollama's `all-minilm` model (configurable via `OLLAMA_EMBED_MODEL`). Each candidate chunk's embedding (if available) is compared via cosine similarity. Chunks without embeddings fall back to a normalized FTS rank scaled by 0.5. Results are sorted by final score and limited to the requested count (default 20 via `SEARCH_DEFAULT_LIMIT`).

This hybrid prioritizes speed for broad indexes (FTS5 scales linearly with content) while improving accuracy for semantic queries (e.g., synonyms or conceptual matches). Embeddings are optional; FTS5 alone suffices if Ollama is unavailable.

Chunks derive from wiki pages generated by the [AI agent](ai-generation.md). During indexing (via `indexFile`), content splits into overlapping segments to preserve context. Configurable parameters include `SEARCH_CHUNK_SIZE` (default 512 characters) and `SEARCH_CHUNK_OVERLAP` (default 64 characters).

The primary API endpoint is `GET /api/search/:wiki?q=<query>&limit=<N>`, requiring authentication. Web routes (`/:wiki?q=<query>`) integrate live search with debounced AJAX results rendered as HTML lists.

## Chunking

Wiki content splits into fixed-size, overlapping chunks via `chunkText` in `src/lib/chunking.ts`. This line-by-line process respects boundaries, avoiding mid-sentence breaks, and carries overlap from prior chunks for context continuity.

```typescript
// From src/lib/chunking.ts
export function chunkText(
  path: string,
  text: string,
  chunkSize = SEARCH_CHUNK_SIZE,
  overlap = SEARCH_CHUNK_OVERLAP,
): Chunk[] {
  const lines = text.split("\n");
  const chunks: Chunk[] = [];
  let current = "";
  let chunkIndex = 0;

  for (const line of lines) {
    if (current.length + line.length + 1 > chunkSize && current.length > 0) {
      chunks.push({ path, chunkIndex, content: current.trim() });
      chunkIndex++;

      // Keep overlap: take the last `overlap` characters of current chunk
      if (overlap > 0 && current.length > overlap) {
        current = current.slice(-overlap);
        // Re-align to line boundary
        const newlinePos = current.indexOf("\n");
        if (newlinePos !== -1) {
          current = current.slice(newlinePos + 1);
        }
      } else {
        current = "";
      }
    }
    current += (current ? "\n" : "") + line;
  }

  // Final chunk
  if (current.trim()) {
    chunks.push({ path, chunkIndex, content: current.trim() });
  }

  return chunks;
}
```

Each chunk stores `path`, `chunk_index` (sequential per file), `content`, and an optional `embedding` BLOB. Overlap ensures adjacent chunks share ~12% context, aiding re-ranking.

## Indexing

After wiki page generation or updates (e.g., via [AI agent](ai-generation.md) or `/api/push`), `indexFile` in `src/lib/indexer.ts` deletes prior chunks for the path, inserts new ones, and asynchronously computes embeddings if enabled.

```typescript
// From src/lib/indexer.ts (simplified)
```

Embeddings batch-compute via `embed` in `src/lib/rag.ts`, serializing Float32Arrays to BLOBs. FTS5 auto-populates via triggers on `wiki_chunks` INSERT/UPDATE/DELETE.

## Search Implementation

The core `search` function in `src/lib/search.ts` orchestrates the hybrid flow.

```typescript
// From src/lib/search.ts (core logic)
export async function search(
  db: Database,
  wikiId: number,
  query: string,
  opts: { limit?: number } = {},
): Promise<SearchResult[]> {
  const limit = opts.limit ?? SEARCH_DEFAULT_LIMIT;
  const FTS_CANDIDATES = 50;

  // Step 1: FTS candidates
  const ftsResults = ftsSearch(db, wikiId, query, FTS_CANDIDATES);
  if (ftsResults.length === 0) return [];

  // Step 2: embed the query and re-rank by cosine similarity
  let queryEmbedding: Float32Array | null = null;
  try {
    queryEmbedding = await embedOne(query);
  } catch {
    // Ollama unavailable — fall back to FTS rank order
  }

  if (!queryEmbedding) {
    // FTS-only fallback (normalized rank)
    // ...
  }

  // Re-rank FTS candidates using vector similarity
  const worstRank = Math.min(...ftsResults.map((r) => r.rank));
  const results: SearchResult[] = [];
  for (const row of ftsResults) {
    const chunkRow = db
      .prepare(
        "SELECT embedding FROM wiki_chunks WHERE wiki_id = ? AND path = ? AND content = ? AND embedding IS NOT NULL LIMIT 1",
      )
      .get(wikiId, row.path, row.content) as { embedding: Buffer } | null;

    let score: number;
    if (chunkRow?.embedding) {
      score = cosineSimilarity(
        queryEmbedding,
        deserializeEmbedding(chunkRow.embedding),
      );
    } else {
      score = normalizeFtsRank(row.rank, worstRank) * 0.5;
    }

    results.push({ path: row.path, chunk: row.content, score });
  }

  return results.sort((a, b) => b.score - a.score).slice(0, limit);
}
```

`ftsSearch` escapes queries:

```typescript
// From src/lib/search.ts
function escapeFtsQuery(query: string): string {
  const cleaned = query.replace(/[^\w\s*]/g, " ");
  const words = cleaned
    .split(/\s+/)
    .filter(Boolean)
    .map((word) => {
      const base = word.replace(/\*/g, "");
      if (!base) return null;
      return word.includes("*") ? `"${base}"*` : `"${base}"`;
    })
    .filter(Boolean);

  if (words.length === 0) return "";
  return words.join(" OR ");
}
```

FTS query joins `wiki_chunks_fts` with `wiki_chunks` on `wiki_id`, using `MATCH` and `ORDER BY rank`.

Cosine similarity (`src/lib/rag.ts`) normalizes dot product:

```typescript
export function cosineSimilarity(a: Float32Array, b: Float32Array): number {
  // Computes dot(a, b) / (||a|| * ||b||)
  let dot = 0;
  let normA = 0;
  let normB = 0;
  for (let i = 0; i < a.length; i++) {
    dot += a[i] * b[i];
    normA += a[i] * a[i];
    normB += b[i] * b[i];
  }
  const denom = Math.sqrt(normA) * Math.sqrt(normB);
  return denom === 0 ? 0 : dot / denom;
}
```

## Web and MCP Integration

Web routes (`src/routes/web.ts`) render search as HTML lists with snippets (150 chars, truncated). A debounced input (500ms) fetches `?q=<query>` via AJAX, populating `.search-results`.

MCP tools (`src/lib/mcp.ts`) expose `search_wiki` for AI agents, formatting results as numbered lists with scores and previews. Supports public and authenticated wikis.

## Design Decisions

- **Hybrid Ranking**: FTS5 handles exact/keyword matches efficiently (O(1) per wiki); vectors catch semantics without full scans.
- **Chunk Granularity**: 512 chars balances precision (file-level too coarse) and overhead (too fine fragments context).
- **Fallbacks**: Robust to Ollama downtime—search degrades gracefully to FTS.
- **Per-Wiki Isolation**: Queries scope to `wiki_id`, enabling multi-tenant scale.
- **No Source Search**: Focuses on curated wiki pages; sources influence generation but are not directly searchable.
